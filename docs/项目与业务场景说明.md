# API-HUB Agent 项目与业务场景说明
# 前置项目
[https://github.com/violet-hekmatyar/API_Hub](https://github.com/violet-hekmatyar/API_Hub)

## 1. 项目定位
**API-HUB Agent** 是在校园 API-HUB 接口平台基础上扩展的智能巡检与诊断模块。

它不是单纯给接口平台加一个聊天助手，而是围绕 API 平台管理者的日常工作设计：帮助管理者定期查看 API 运行状态、结合校园事件提前关注可能受影响的接口、分析异常原因，并生成简洁的处理建议。

校园是本项目的主要落地场景。API-HUB 统一管理学校相关 API，例如统一登录、今日课表、讲座活动、校园通知、场地预约等接口，并开放给校内小程序、社团系统、课程助手、活动报名系统等应用使用。

---

## 2. 场景背景
原 API-HUB 平台已经具备接口发布、网关转发、签名鉴权、调用统计、限流降级、SDK 调用等能力。

随着接入 API 和调用方增加，平台管理者会遇到新的问题：

| 问题 | 说明 |
| --- | --- |
| API 状态需要持续关注 | 统一登录、课表、讲座报名等接口对校内应用影响较大 |
| 校园事件会影响接口访问 | 开学、讲座报名、考试安排、活动签到等会带来短时间访问变化 |
| 异常原因分散 | 需要同时看调用统计、网关日志、错误码、限流记录和接口文档 |
| 处理建议需要整理 | 管理者需要判断是联系调用方、通知接口提供者，还是调整限流 / 缓存策略 |
| 问题需要复盘 | 一天或一周结束后，需要知道发生过哪些高并发事件、哪些 API 风险较高 |


因此，Agent 的价值不是替代后端规则，而是把已有平台数据和文档能力组织起来，帮助管理者做巡检、预警、诊断和总结。

---

## 3. 典型 API 类型
| API 类型 | 示例 | 可能关注的问题 |
| --- | --- | --- |
| 统一身份类 | 学校账户授权登录、Token 刷新 | 401 / 403、签名失败、Token 过期 |
| 教务信息类 | 今日课表、考试安排、教学周信息 | 开学高峰访问、缓存压力、响应变慢 |
| 校园活动类 | 讲座活动、活动报名、签到状态 | 报名开放时访问集中、超时、限流 |
| 通知公告类 | 学院通知、社团通知、校园公告 | 通知发布后短时间访问增加 |
| 资源预约类 | 场地预约、实验室预约、设备借用 | 预约开放时并发冲突、重复请求 |

## 4. 主要用户
## 4.1 平台管理者：核心用户
平台管理者是 API-HUB Agent 的主要使用者。

| 使用场景 | 示例问题 |
| --- | --- |
| 事件前预警 | 本周有哪些校园活动可能影响 API 调用？ |
| 日常巡检 | 今天有哪些 API 状态需要关注？ |
| 异常诊断 | 为什么统一登录 API 的 403 变多了？ |
| 接口复盘 | 本周发生了多少次高并发事件？ |
| 处理建议 | 讲座报名 API 是否需要提前限流或增加缓存？ |


---

## 4.2 API 提供者：问题协同对象
API 提供者不是 Agent 的第一入口。管理者发现问题后，可以让 Agent 整理一份面向接口提供者的问题摘要。

例如：

```plain
请把讲座报名 API 昨天的异常情况整理给接口提供者。
```

Agent 可以汇总异常时间段、错误码、慢请求、调用方分布和建议处理方式，减少管理者手动整理沟通材料的成本。

---

## 4.3 API 使用者：辅助使用场景
API 使用者可以使用部分 Agent 能力进行自助排错，但不是项目主线。

例如：

```plain
我调用今日课表 API 返回 403，帮我看一下可能原因。
```

Agent 可以查询错误码说明、签名规则和最近调用日志，给出排查方向。

这类能力主要是管理者治理能力的延伸，用来减少重复答疑。

## 5. 核心业务流程
## 5.1 事件预警流程
事件预警发生在业务事件之前，重点是提前判断哪些 API 可能受到影响。

```plain
读取校园事件
→ 提取事件时间、事件类型、影响对象
→ 匹配可能受影响的 API
→ 查询相关 API 的历史调用情况
→ 生成预警建议
```

示例：

| 校园事件 | 可能受影响 API | Agent 预警建议 |
| --- | --- | --- |
| 周四 12:00 讲座报名开放 | 统一登录 API、讲座活动 API、报名 API | 报名前后重点观察调用量、失败率和 P95 响应时间 |
| 开学第一周课表更新 | 统一登录 API、今日课表 API | 提前检查课表接口缓存和限流配置 |
| 考试安排发布 | 考试查询 API、校园通知 API | 关注短时间集中访问和响应变慢风险 |
| 活动现场签到 | 统一登录 API、活动签到 API | 关注现场网络波动导致的重复请求和签名失败 |


这里的重点不是“调用量升高就报警”，而是结合事件上下文提前提醒管理者需要关注哪些接口。

---

## 5.2 日常巡检流程
日常巡检发生在运行过程中或事件结束后，重点是查看 API 实际运行情况。

```plain
定时任务触发
→ 查询核心 API 调用统计
→ 查询失败率、响应时间、限流记录
→ 查询网关日志样例
→ 结合当天已知事件进行解释
→ 生成巡检摘要
```

示例输出：

```plain
今日 API 巡检摘要：

1. 统一登录 API 整体运行正常，但 10:00-11:00 出现少量 403，主要来自课程助手应用，疑似签名配置问题，建议联系调用方确认密钥配置。

2. 讲座报名 API 在 12:00-12:30 出现访问高峰，与今日讲座报名开放时间一致。接口未出现明显失败率升高，但 P95 响应时间上升，建议后续类似活动提前增加缓存或限流观察。
```

巡检不只是发现异常，也可以记录“符合预期但需要关注”的事件，为后续复盘提供依据。

---

## 5.3 异常诊断流程
当某个接口出现明显异常时，管理者可以主动询问 Agent。

```plain
管理者提问
→ Agent 判断问题类型
→ 查询调用统计
→ 查询错误码分布
→ 查询网关日志
→ 检索接口文档和签名规则
→ 汇总原因和建议
```

示例问题：

```plain
为什么今天统一登录 API 的 403 变多了？
```

示例结论：

```plain
统一登录 API 今日 403 错误主要集中在 10:00-11:30，异常调用方为课程助手应用。
从网关日志看，主要错误为签名校验失败，而不是用户权限不足。
结合签名规则文档，可能原因是调用方 accessKey / secretKey 未更新，或请求时间戳与服务端时间偏差过大。
建议联系调用方维护者检查密钥配置，并继续观察该应用后续 1 小时失败率变化。
```

---

## 5.4 周报 / 复盘流程
周报用于对一段时间内的 API 状态做总结，适合管理者复盘平台稳定性。

```plain
统计一周 API 运行数据
→ 汇总高并发事件
→ 汇总异常接口
→ 汇总高频错误码
→ 统计重点调用方
→ 生成复盘报告
```

示例输出：

```plain
本周 API 运行复盘：

1. 本周共发生 3 次高并发访问事件，分别与讲座报名、考试安排发布、活动签到有关。
2. 统一登录 API 是本周调用量最高的接口，整体成功率稳定，但存在两次集中 403。
3. 讲座报名 API 在报名开放后 30 分钟内 P95 响应时间明显上升，但未触发熔断。
4. 今日课表 API 在开学相关访问高峰中表现稳定，缓存策略有效。
5. 建议下周重点关注活动报名类接口，并提前为统一登录 API 设置更细粒度的调用方监控。
```

## 6. 技术方案概览
## 6.1 总体链路
```plain
定时任务 / 管理者提问 / 业务事件触发
→ Agent 理解任务类型
→ 选择调用统计、日志、文档、限流规则、事件查询等工具
→ 必要时通过 RAG 检索接口文档、错误码和签名规则
→ 汇总证据
→ 生成巡检报告、事件预警、异常诊断或周报
→ 通过 SSE 返回分析过程和最终结果
```

---

## 6.2 核心工具设计
| Tool | 作用 | 数据来源 |
| --- | --- | --- |
| `queryApiCallStats` | 查询调用量、成功率、失败率、平均耗时、P95 耗时 | API-HUB 调用统计 |
| `queryGatewayLogs` | 查询签名失败、参数错误、路由失败、限流记录 | 网关日志 / Mock 日志 |
| `queryApiDocs` | 查询接口文档、错误码、签名规则、SDK 说明 | RAG 知识库 |
| `queryRateLimitRules` | 查询限流、熔断、降级配置 | Sentinel / 平台配置 |
| `queryConsumerProfile` | 查询调用方应用、负责人、授权状态、调用频率 | API-HUB 应用信息 |
| `queryCampusEvents` | 查询讲座、开学、考试、活动报名等校园事件 | 事件日历 / Mock 数据 |
| `generateReport` | 生成巡检报告、异常摘要、通知模板 | Agent 汇总结果 |

## 7. 技术点设计
项目不追求一次性塞满所有 Agent 技术，而是按业务需要逐步加入。

## 7.1 第一版核心能力
| 技术点 | 在项目中的作用 |
| --- | --- |
| Agent | 根据管理者问题判断是巡检、预警、诊断还是周报 |
| Tool Calling | 调用统计、日志、限流规则、事件查询等后端能力 |
| RAG | 检索接口文档、错误码、签名规则和 SDK 说明 |
| SSE | 流式展示分析过程和最终报告 |
| 结构化输出 | 输出风险等级、证据、建议动作、通知摘要 |
| Prompt 约束 | 限定 Agent 只处理 API 管理和诊断相关任务 |
| 工具失败兜底 | 工具失败时说明缺失信息，不编造结论 |


---

## 7.2 测试版本 / 探索性增强
这些能力不一定作为第一版核心功能，但可以以测试版本或后续优化的形式体现。

| 技术点 | 使用方式 |
| --- | --- |
| Tool Trace | 记录工具调用参数、结果、耗时和错误，方便复盘 Agent 判断过程 |
| 引用展示 | 报告中标注参考的接口文档、错误码说明或校园事件 |
| 小型评测集 | 构造常见异常问题，验证工具选择和 RAG 召回是否正确 |
| 工具协议设计 | 借鉴 MCP 的思想，统一工具名称、参数 schema、返回结构和错误格式 |
| 混合检索 | 对错误码、接口名等精确字段增加关键词检索 |
| Rerank | 当文档规模变大后，对 Top-K 召回结果做重排序 |
| 多 Agent | 后续可拆成巡检 Agent、诊断 Agent、报告 Agent，但第一版不强行拆分 |
| 人工确认 | 对限流调整、通知调用方等高风险动作只生成建议，不自动执行 |

## 8. 为什么需要 Agent，而不是普通规则系统？
普通规则系统适合处理明确条件，例如：

```plain
如果某 API 失败率超过 10%，则报警。
```

但 API-HUB 的管理问题经常需要结合多个信息源判断。

| 问题 | 为什么 Agent 更合适 |
| --- | --- |
| 访问量上涨是否异常 | 需要结合校园事件和历史数据判断 |
| 403 增加是什么原因 | 需要查日志、签名规则、权限状态和调用方信息 |
| 是否需要联系 API 提供者 | 需要判断问题来自接口实现、文档缺失还是调用方错误 |
| 是否要提前关注某接口 | 需要结合活动时间、接口类型和历史高峰 |
| 如何写复盘报告 | 需要把统计、日志、文档和事件信息组织成可读结论 |


因此，Agent 的定位是：在规则和报表之上，做多源信息综合、解释和建议生成。

---

## 9. 稳定性与可持续性设计
API-HUB 是基础平台，Agent 不能破坏原有接口调用链路。

| 原则 | 说明 |
| --- | --- |
| Agent 作为增强层 | 不直接阻塞原 API 调用，只读取统计、日志、配置和文档 |
| 高风险动作人工确认 | 限流、下线、通知调用方等动作只生成建议，不自动执行 |
| 工具协议稳定 | 每个工具有明确输入、输出和错误格式，方便后续扩展 |
| Mock / Local / Prod 分离 | 开发阶段使用 Mock 数据，后续可替换真实数据源 |
| 兼容后续 API 扩展 | 新增 API 时通过 metadata 和统一工具协议接入分析链路 |
| 保留工具调用记录 | 方便复盘 Agent 为什么给出某个结论 |

## 10. 项目一句话介绍
**API-HUB Agent 是一个面向校园 API-HUB 平台管理者的智能巡检与诊断模块。它基于 Agent、RAG、Tool Calling 和 SSE，将接口文档、错误码、签名规则、调用统计、网关日志、限流配置和校园事件统一到一个分析流程中，支持事件前预警、日常巡检、异常诊断和周报复盘，帮助管理者更快发现 API 风险并生成可解释的处理建议。**
